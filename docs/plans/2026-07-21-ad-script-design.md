# 口播脚本重新实现设计

## 背景与目标

现有 `/aigc/ad-script` 使用通用 `ModulePage` 和通用 Worker 生成链路，只能提交少量字段并展示通用文本结果。新实现参考 `/Users/jik1992/Desktop/公司/口播脚本_产品设计稿` 的完整产品流程，但视觉上遵循烽火AI现有 `DESIGN.md`，不复制参考稿的高饱和蓝色设计语言。

目标是交付完整真实功能：场景选择、已有脚本解析、产品和广告诉求、脚本风格、1–3 条 A/B 生成、最多 5 轮 AI 调优、通用广告合规检查、版本管理、人工编辑、评分、复制与 TXT/Markdown 导出。

## 已确认的产品决策

- 实现独立口播脚本业务页面、专用 OpenAPI 接口和专用 Worker Handler，同时复用现有鉴权、BullMQ、SQLite Job、SSE、创作点和通知基础设施。
- 全部模型阶段强制使用 `deepseek/deepseek-v4-pro`，页面不提供模型选择。
- 模型不可用、未验证或调用失败时明确失败，不静默切换其他模型或 Mock。
- 合规检查只覆盖平台无关的广告法风险、极限词、违禁承诺、无法证明的功效表达、字数与 CTA 结构，不绑定投放平台。
- 每条脚本最多调优 5 轮。总分达到 85 且合规通过时提前结束。
- 支持一次生成 1–3 条 A/B 脚本，每条独立生成、调优和比较。
- 支持粘贴已有口播或竞品脚本，由模型提取结构化产品信息；只有用户确认后才写入表单。
- 结果正文可直接编辑并保存为人工版本；后续评分、继续调优和导出均以用户选定版本为输入。
- 转化力评分采用四个等权维度：开场吸引力、痛点共鸣度、利益点清晰度、行动召唤强度，各占 25 分。合规作为独立门槛。
- 每条脚本固定消耗 20 创作点。批量任务只要至少一条成功就不退款；整批完全失败时全额退款。
- 导出支持复制全文、TXT 和 Markdown。

## 整体架构

### Web

`/aigc/ad-script` 改由独立 `AdScriptPage` 渲染，仍位于现有 `AppShell` 中。页面使用生成的 API SDK 访问服务端，不手写重复 DTO 或请求路径。

页面分为：

1. 场景与批量数量。
2. 产品信息与广告诉求。
3. 演讲角色和脚本风格。
4. AI 调优实时状态。
5. 结果编辑、评分分析、版本和导出。

### API Server

新增口播脚本专用 OpenAPI 路由，负责：

- 保存和读取项目草稿。
- 解析用户粘贴的已有脚本。
- 创建批量生成任务并一次性扣费。
- 查询项目、变体、版本、评分和合规结果。
- 保存人工编辑版本。
- 对指定版本重新评分、继续调优或重新生成。
- 输出 TXT 和 Markdown 文件。

全部接口经过 Bearer Token 鉴权，服务端从鉴权上下文取得 owner，不接受前端提交 owner ID。

### Worker

新增专用 `worker/jobs/job-ad-script.ts` 并在通用 fallback 之前注册。BullMQ 消息继续只携带 `jobId`，Worker 从 SQLite 获取持久状态。

每个变体执行：

```text
构建结构化上下文
→ 生成初稿
→ 四维评分
→ 确定性合规规则检查
→ 未达标时按问题改写
→ 最多重复 5 轮
→ 保存最终版本
```

每个阶段先持久化结果再推进状态，使页面刷新、Worker 重启、重复投递和任务恢复不会丢失进度或重复执行已完成阶段。

## 数据模型

新增三类 Drizzle 表，并通过新 migration 演进：

### `ad_script_projects`

保存 owner、输入草稿、场景类型、场景 ID、营销目标、目标用户、痛点、利益点、角色、风格、开场方式、批量数、整体状态、Job 关联和时间戳。

### `ad_script_variants`

保存所属项目、变体序号、状态、当前版本、最终得分、合规状态、实际迭代次数和错误摘要。项目与序号建立唯一约束，作为重复投递时的幂等边界。

### `ad_script_versions`

保存所属变体、顺序号、来源（初稿、AI 调优、人工编辑）、父版本、轮次、脚本文本、四维评分、合规结果、修改摘要、模型和时间戳。版本不可覆盖，`currentVersionId` 指向当前选定版本。

结构化输入、评分明细和合规命中可以使用经 Zod 校验的 JSON 字段；核心关联、状态、顺序和所有权保持为独立列。

## 页面与交互

### 第一步：场景与数量

- “营销场景 / 投放场景”标签切换。
- 场景卡片单选，展示适用策略和推荐字数。
- A/B 数量为 1–3，实时展示 `20 × 数量` 的预计费用。

### 第二步：产品信息与广告诉求

- 可展开已有脚本解析区，支持粘贴、解析、预览提取字段和确认填入。
- 产品名称、最多 6 条产品卖点和目标字数。
- 营销目标、目标用户、用户痛点和产品利益点。
- 必填项即时校验，切换步骤时保留草稿。

### 第三步：脚本风格

- 演讲角色：好物推荐员、KOL 达人、普通用户、行业专家、品牌官方和自定义角色。
- 脚本风格、开场方式与可折叠高级设置。
- 提交前汇总任务数量、费用、固定模型和预计等待时间。

### AI 调优状态

- 展示每条 A/B 变体的独立状态和整体进度。
- 每轮展示初稿、评分、改写、合规与最终评分。
- 从服务端恢复真实状态，不使用前端假进度。
- 支持取消整批任务，不支持只取消单个变体。

### 结果页面

- 顶部切换 A/B 变体，展示总分、字数、迭代次数和合规状态。
- 左侧为可编辑脚本，保存后生成新的人工版本。
- 右侧展示四维评分、命中原句、修改建议和版本时间线。
- 支持重新评分、继续调优、重新生成、复制、TXT 和 Markdown 导出。
- `1440×900` 使用双栏，`1024×768` 改为上下布局并保持关键操作可见。

## 模型与结构化输出

所有生成、解析、评分、语义合规补充和改写阶段固定使用 `deepseek/deepseek-v4-pro`。模型标识集中在服务端口播脚本定义中，不由浏览器提交。

模型必须返回符合 Zod Schema 的结构化 JSON。解析失败时允许有限次数的同模型修复；仍无法通过时返回 `MODEL_OUTPUT_INVALID`，不得接受未经验证的自由文本。

评分模型只返回每个固定维度的分值和证据，Server 校验 0–25 范围并重新计算总分。模型不得直接控制最终总分或合规通过状态。

## 合规检查

确定性本地规则至少覆盖：

- 极限词和绝对化表达。
- 保证性、承诺性或无法证明的效果表达。
- 典型违禁或高风险广告词。
- 目标字数范围。
- CTA 是否存在且与场景一致。

每条命中返回规则 ID、严重级别、原句范围、说明和修改建议。DeepSeek 可以补充语义风险，但不能消除本地规则命中。只有不存在阻断级命中时才视为合规通过。

## 状态、计费与恢复

创建任务时，Server 在一个数据库事务中校验输入、创建项目和变体、扣除 `20 × 数量` 创作点并创建 Job。入队失败时保留可恢复状态，由现有恢复流程重新入队。

- 使用唯一幂等键避免重复提交、重复版本和重复扣费。
- 整批完全失败时通过账本唯一键执行一次全额退款。
- 至少一个变体成功时不退款。
- 用户取消不退款，并保留已经完成的版本。
- 人工版本永不覆盖 AI 版本；重新评分和继续调优必须引用明确的输入版本 ID。

## 错误处理

所有接口返回项目统一错误结构：`code`、`message`、`retryable` 和 `requestId`。

主要错误包括：

- `MODEL_NOT_AVAILABLE`：固定模型未验证或不可用。
- `MODEL_OUTPUT_INVALID`：模型结构化输出多次修复失败。
- `INSUFFICIENT_CREDITS`：创作点不足，保留草稿。
- `COMPLIANCE_RULE_ERROR`：本地合规检查异常，禁止伪装为通过。
- `PARTIAL_BATCH_FAILURE`：部分变体失败且不退款。
- `BATCH_FAILED_REFUNDED`：整批失败并已执行幂等退款。
- `VERSION_CONFLICT`：人工编辑基于过期版本。

Worker 或 Redis 临时故障依靠 SQLite 状态恢复，不重复扣费。模型失败不切换 Provider、模型或 Mock。
