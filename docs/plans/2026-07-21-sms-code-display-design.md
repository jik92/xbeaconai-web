# 开发阶段短信验证码回显设计

## 目标

项目尚未接入真实短信服务。用户点击“获取验证码”后，除了保留服务端日志输出，还需要在注册和忘记密码页面直接显示本次验证码，以便完成开发和验收流程。

当前阶段所有运行环境都回显验证码，后续接入短信服务时再移除响应字段和页面展示。

## 数据流

`AccountStore.sendSmsCode` 生成并保存验证码哈希、调用短信发送器后，将明文验证码作为发送结果返回。`POST /api/auth/sms-code` 响应增加必填字段 `verificationCode`，OpenAPI 文档和生成的前端 SDK 同步更新。

前端认证上下文返回完整的验证码发送结果。注册页和忘记密码页请求成功后，将 `verificationCode` 保存到当前认证页面状态，并显示“当前验证码：xxxxxx”。重新发送时覆盖旧验证码。

## 交互与状态清理

- 验证码发送成功后，在验证码输入框下方显示当前验证码。
- 保留 60 秒重新发送倒计时。
- 切换登录、注册、忘记密码或设置密码页面时清空已显示验证码。
- 发起新的验证码请求时先清空旧验证码，避免请求失败后继续显示过期值。
- 注册和忘记密码两种用途使用同一展示方式。

## 安全边界

验证码数据库仍只保存密码哈希，服务端日志行为保持不变。明文验证码会通过 API 返回给任何请求方，这是当前开发阶段明确接受的行为，不应视为生产安全方案。

接入真实短信服务时必须同时移除：

1. API 响应中的 `verificationCode`。
2. 前端认证上下文中的对应字段。
3. 页面上的验证码回显区域。

## 测试与验收

- 单元测试验证短信发送结果包含本次生成的验证码，但数据库不保存明文。
- API/OpenAPI 契约验证 `verificationCode` 为 6 位数字字符串。
- E2E 验证注册和忘记密码页面都能显示当前验证码，并能使用该验证码完成验证。
- 运行格式检查、单元测试、类型检查、生产构建和桌面/Tablet E2E。
