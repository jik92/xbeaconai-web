# 口播脚本一分钟生成与可观测性设计

## 目标

口播脚本自动生成应以一分钟内完成为目标，后台必须能看到不包含脚本正文或密钥的阶段日志。后续人工触发的“继续调优”也应在单次操作内快速完成，同时保留累计最多五轮的产品约束。

## 执行方案

- 将“生成/改写”和“四维评分/语义合规”合并为同一次 DeepSeek 结构化输出，避免每个版本连续调用模型两次。
- 首次生成最多执行初稿和一轮自动改写，共两次模型调用；未达 85 分或仍不合规时由结果页提示继续调优。
- 每次点击“继续调优”只推进一轮，累计轮次仍不得超过五轮。
- 1–3 个 A/B 变体通过 `Promise.allSettled` 并行执行，单个变体失败不阻断其他变体。
- 为自动生成和继续调优设置 55 秒截止时间。Provider 请求使用剩余预算作为超时，超时返回明确的可重试错误，不无限等待。

## 日志与界面

- Worker 使用单行 JSON 结构化日志记录任务 ID、变体、轮次、阶段、模型、耗时、得分、合规状态和错误码。
- 不输出 Prompt、脚本正文、Token 或 Provider 密钥。
- 进度页显示真实已耗时和“一分钟内完成”的目标说明，继续使用 SQLite 状态而非前端假进度。

## 验证

- 单元测试覆盖合并结构化输出、55 秒预算、单次继续调优只增加一轮及最大五轮限制。
- Worker 测试覆盖 A/B 并行和部分失败。
- Playwright 覆盖进度页耗时与目标提示。
- 使用真实 `deepseek/deepseek-v4-pro` 测量一次完整生成耗时，并记录结果。
