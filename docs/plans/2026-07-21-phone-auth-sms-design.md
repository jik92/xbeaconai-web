# 手机号认证与日志短信验证码设计

## 目标

将开发阶段现有的邮箱认证完整替换为手机号认证：用户使用手机号、短信验证码和密码注册，之后使用手机号和密码登录。短信服务暂不连接第三方平台，验证码由服务端日志输出，同时保留稳定的发送器接口供后续替换。

默认管理员手机号通过服务端环境变量 `ADMIN_PHONE` 配置，本地默认值为 `17688743518`。

## 认证流程

### 注册

1. 用户填写中国大陆手机号并请求验证码。
2. 服务端校验手机号、发送频率和账号占用状态。
3. 服务端生成 6 位数字验证码，将哈希写入 SQLite，并通过日志短信发送器输出明文验证码。
4. 用户提交手机号、验证码、显示名称和密码。
5. 服务端校验验证码的用途、有效期、错误次数和消费状态。
6. 在同一业务流程中消费验证码、创建账号、初始化默认素材目录和偏好，然后签发登录令牌。

### 登录

用户使用手机号和密码登录。密码规则保持至少 10 位，并同时包含字母和数字。错误响应不泄露手机号是否存在。

## 验证码模型

新增短信验证码表，保存手机号、用途、验证码哈希、过期时间、错误次数、消费时间和创建时间。验证码规则：

- 6 位数字。
- 5 分钟有效。
- 同一手机号 60 秒内不能重复发送。
- 仅允许用于注册。
- 验证成功后立即消费，不能重复使用。
- 达到最大错误次数后作废。

当前 `ConsoleSmsSender` 使用结构化服务端日志输出手机号、用途、验证码和过期时间，不记录密码或认证令牌。后续短信平台实现相同发送器接口即可替换。

## 数据迁移

`users.email` 替换为唯一的 `users.phone`，用户摘要、资料更新、任务管理筛选和所有 API 契约统一使用手机号。

不删除已有用户关联的任务、素材和生成物。已有开发账号迁移为带内部遗留标识且不可登录的禁用用户，避免伪造可用手机号；新手机号需要重新注册。

新增管理员环境变量：

```dotenv
ADMIN_PHONE=17688743518
```

该变量只在服务端读取。服务端在用户摘要中返回管理员标识，前端据此显示和保护管理入口，不把管理员手机号编译进前端资源。

## API 契约

- `POST /api/auth/sms-code`：接收手机号，发送注册验证码。
- `POST /api/auth/register`：接收 `phone`、`verificationCode`、`displayName` 和 `password`。
- `POST /api/auth/login`：接收 `phone` 和 `password`。
- 用户摘要将 `email` 替换为 `phone`，并增加服务端计算的 `isAdmin`。
- 管理员任务列表的邮箱筛选改为手机号筛选。

所有接口继续使用结构化错误格式，覆盖手机号格式错误、手机号已注册、验证码错误、验证码过期、验证码尝试次数耗尽和频率限制。

## 前端交互

登录页默认显示手机号和密码。注册页显示昵称、手机号、验证码、获取验证码按钮和密码。

- 手机号格式无效时不允许请求验证码。
- 获取成功后按钮进入 60 秒倒计时。
- 页面提示用户从服务端日志查看当前开发验证码，但不在浏览器响应中回显验证码。
- 登录和注册共享现有紧凑认证卡片、输入框、密码显隐和错误提示样式。
- 管理后台、任务筛选和个人资料中的邮箱文案与字段全部替换为手机号。

## 安全与并发

- 数据库只保存验证码哈希，日志输出明文仅用于当前开发阶段。
- 发送频率同时按手机号和客户端来源限制。
- 验证失败递增尝试次数，成功消费使用事务性更新，避免并发重复注册。
- 密码继续使用 Bun 密码哈希。
- 注册、登录和验证码接口均保留速率限制。

## 测试与验收

- 验证码生成、哈希、过期、错误次数、冷却和单次消费单测。
- 手机号注册、重复注册、错误验证码和手机号密码登录测试。
- 登录注册页面 E2E，覆盖验证码请求和倒计时。
- 管理员手机号识别、任务手机号筛选和普通用户权限隔离测试。
- 生成 Drizzle migration、OpenAPI 文档和前端 SDK。
- 运行 Biome、单测、类型检查、生产构建、数据库迁移检查和完整 E2E。
