# 一键成片脚本与分镜操作设计

## 问题

一键成片右侧已有分镜时，脚本顶部“重新生成”仍会先保存左侧参数。项目处于 `storyboard_review` 状态时，服务端拒绝该参数更新并返回 422。左侧主按钮还会在脚本生成后切换成“生成分镜”，把两个阶段混在同一个入口中。

## 方案

采用限制式流程，不扩大服务端可编辑状态：

- 左侧底部按钮只执行脚本生成。没有脚本时显示“生成脚本”，已有脚本但尚无分镜时显示“重新生成脚本”。
- 右侧脚本结果区底部提供“生成分镜”，负责从已确认脚本进入分镜阶段。
- 一旦项目已经存在分镜，左侧脚本按钮和脚本顶部“重新生成”都禁用，避免旧分镜与新脚本不一致。
- 已有分镜时，右侧按钮显示“分镜已生成”并禁用。用户继续在“分镜”页生成或替换镜头，不从脚本页重新建立分镜。
- 单段“换一版”和脚本文本保存同样在已有分镜后禁用，避免局部脚本变更使既有分镜失效。

## 状态与错误处理

按钮是否可用由项目是否已有 `shots` 决定，不依赖易漂移的前端状态文案。任务执行中沿用现有 `busy` 和项目状态禁用逻辑。前端不再从已有分镜的项目发出脚本参数 PATCH，因此不需要放宽服务端状态机。

## 验证

- 单元测试检查左侧入口始终提交 `script`，不再根据脚本数量提交 `storyboard`。
- 单元测试检查已有分镜时，全量脚本重新生成、单段换版和文本保存入口都被禁用。
- 单元测试检查右侧脚本区承担“生成分镜”操作。
- 运行相关单测、类型检查和生产构建；E2E 留给用户手动验证。
